Date: 2026-07-06. Follow-up to `docs/2026-07-06-full-studio-vision-research.md` (which named and closed the three concrete production-pipeline gaps: character-consistency, post-production slice 1, reverse adaptation). This pass covers three things the user asked for together: (1) enhancement of already-built features, found by auditing GhostWriter's own code, not guessing; (2) new features that plug into the existing capability-registry/role architecture rather than requiring a new pillar, informed by live competitor research; (3) deeper scoping for the showcase area.

## 0. Graphify-assisted follow-up (same day)

The user asked for a graphify-driven pass over the whole codebase before acting on the sections below. `graphify query` surfaced `docs/superpowers/underused-features-backlog-2026-06-29.md` — a verified 11-item underused-features backlog from a week earlier. Re-checking each item against the current code found **most of it has since been resolved** by work already logged in CLAUDE.md: Reader Insights (`ReaderInsightsPanel.tsx`, wired into `ExportStageView`), the Series/Universe dashboard shelf (`UniverseShelf.tsx`/`CreateUniverseModal.tsx`), the Admin Ops dashboard (`src/app/admin/page.tsx` — cost/analytics/cache-stats, all three), and the Craft Library panel are all built and reachable now, contrary to the older doc's "❌ not surfaced" marks.

**One genuinely new, high-confidence finding came out of this re-check**: `storyThreads`/`storyPromises`/`storyCheckpoints` (`src/db/schema.ts:399-448`) are a **complete, fully-built CRUD backend with zero UI anywhere**. `POST /api/projects/[projectId]/story-state` creates/updates named story threads (subplot/status/open-close chapter refs) and promises (setup text + chapter, payoff intent + chapter, priority A/B/C, status) — a genuine structured "planted vs. paid-off" tracker, hand-authored by the user rather than LLM-inferred. `checkpoints/route.ts` + `storyCheckpoints` snapshot word/chapter counts and open-thread/promise counts at a named milestone. Neither table is referenced by a single `.tsx` file — confirmed via grep, not inferred. This is a stronger candidate than the LLM-based "continuity auditor" idea in §2 below: it needs **zero new AI/prompt design**, just a UI panel calling routes that already exist and work. The existing `promise-ledger.ts` (used in `refine`/`prose-fix`) is a completely separate, LLM-inferred mechanism — it does not read these tables at all, so wiring up a UI here would surface a *second*, user-authored source of truth alongside the automatic one, not duplicate it.

## 1. Enhancement opportunities in what's already built

Verified directly against the code, not the changelog's prose summary:

- **Story Graph capability-wiring has real headroom.** `NODE_CAPABILITIES` (`src/lib/graph/graph-program.ts:29`) wires only 10 of the registry's ~52 total capabilities to graph nodes, and two of the five node kinds have **zero** wired capabilities: `location` and `world_entity` both map to `[]`. Dragging a wire from a location or world-entity node onto anything currently does nothing — there's no capability for it to reach. Concrete candidates once picked up: a location node wired to a "generate location reference image" capability (reusing Production Studio's existing image-gen path), or a world-entity node wired to whatever eventually becomes the continuity-audit capability below.
- **`series-plan`/`research-scaffold` are reachable but siloed.** Both have real routes (`/api/ai/series-plan`, `/api/ai/research-scaffold`) and UI (`ResearchScaffoldPanel.tsx`, surfaced in `WorldBiblePanel.tsx`) — they're not dead, just intentionally excluded from the RAG-context wiring pass (item 28, "creator content-planning, not fiction-narrative tools") and from the Story Graph. Worth a deliberate decision on whether they stay creator-tool-only or eventually get a fiction-facing equivalent (a "story bible for series bibles" angle), rather than leaving it as an unexamined default.
- **Mode Registry core/experimental split is still the one item left from the architectural-debt note.** CLAUDE.md's own "Known architectural debt" section lists "a `MODE_REGISTRY` core/experimental split" as a planned-but-not-done consolidation step, alongside zod-typed JSONB (done) and the Discover/Shape/Write/Produce restructure (done via the funnel work). 26 modes with no core/experimental tagging means the density-dial's Simple/Standard/Full tiers (`src/lib/modes/density.ts`) are the only thing currently doing this job, informally.
- **Headroom has no v4.** v1 (lossless compaction) → v2 (priority packing) → v3 (LLM summarization) are all shipped and wired into the main `generate()` path only — the other 19 library-mode generators in `useGeneration.ts` don't get Headroom's overflow-rescue at all (deliberately, per item 36 — "lower-traffic and Opus-tier"). Worth re-checking that assumption once real usage data exists, rather than treating it as permanent.
- **Video previz is already half-built and underexposed.** Production Studio's preview→animate two-stage flow (generate a still Soul image, then animate it into video) is structurally the same "cheap draft before expensive final" pattern the 2026 competitor landscape has converged on (Midjourney's Draft Mode, multi-model routing between a fast/cheap model and a premium "hero" render — see §2 below). GhostWriter already has the two stages; it just doesn't yet have a *cheap draft-tier for the video generation itself* (e.g. a fast/short/low-res Seedance pass before committing to the full-duration, full-reference-image final render). This is a genuine enhancement, not a new feature — extending the existing gate rather than adding one.
- **Comic Studio's own backlog items are still open**: the storydiffusion swap (built, opt-in, cost-unvalidated — needs the ~$10 real-money pass already logged in memory) and multi-shot Seedance restructuring (explicitly dropped from scope by the user earlier this session, still on the shelf if picked back up later).

## 2. New features that plug into existing architecture (not a new pillar)

Researched live what 2026's competitor landscape is converging on for fiction-writing tools specifically (not the media-generation side, which the prior research doc already covered), then checked what of it GhostWriter could build as an *extension* of things it already has — the Director/Writer/Editor role split (`src/lib/roles/`), the capability registry, the quality-check Tier-1/Tier-2 stack (`useGeneration.ts:178`, gated by `qualityGradingEnabled`), World Bible, and the promise-ledger.

- **Multi-persona "beta reader panel."** A real, fast-growing 2026 category: Slima (named virtual readers with DNF triggers and an "Analyst"/"Intuitive" persona split), Beta Reader AI (multiple LLMs voting independently then a consensus pass), AutoCrit and Storysnap (custom reader personas, 5 simultaneous reads compared). Ideorix — GhostWriter's closest direct competitor per the prior research — ships exactly this as its "5-lens Beta Reader Panel." GhostWriter's existing quality-check stack (Haiku Tier 1 + Sonnet Tier 2, non-blocking, `qualityCheck` flag on `MODE_REGISTRY` entries) is structurally most of the way there — it's already a "multiple graders read this and report back" pipeline, just with generic craft-quality graders instead of named reader personas with distinct sensitivities. Turning it into a beta-reader panel is mostly a prompt/persona-definition change plus a result-aggregation UI, not new plumbing — a strong "already-built architecture, new feature" candidate.
- **Manuscript-wide continuity/contradiction audit.** 2026 competitor research (Novarrium, CipherWrite's "Manuscript Auditor," epos-ai) converges on the same requirement: continuity checking needs "persistent memory with a structured project database" checked against the whole manuscript, not just recent text — otherwise tools "catch explicit timeline contradictions but miss vague ones." GhostWriter already has exactly that structured database: World Bible (characters/locations/plot-threads/world-entities with typed fields) plus the promise-ledger (`src/lib/ai/promise-ledger.ts`, unresolved-promise aggregation) plus Story Graph health probes (`analyzeGraphHealth` — isolated characters, unrooted threads). A continuity-audit capability would cross-reference generated prose against these existing structured records (e.g., flag a chapter that gives a character a described trait contradicting their World Bible entry) — again an extension of Editor-role tooling plugging into data structures that already exist, not a new database.
- **Genre-specific craft rules.** Ideorix ships 70+ genre-specific rule sets in its editorial pipeline. GhostWriter's `MODE_REGISTRY` already has a `realismDomains` field and per-mode context policy — extending that with genre-tagged craft directives (similar in shape to the existing `WRITE_CRAFT_DIRECTIVES` constant in `engine.ts`) is additive to a field that already exists, not a new mechanism.
- **A cheap video draft-tier**, per §1 above — same conclusion from the competitor side: "the practical workflow most creators land on is drafting in a Fast/Lite model, then re-rendering the survivors in Quality," saving 30-50% in reported cost comparisons. Maps directly onto Segmind's existing model list (`src/lib/higgsfield/models.ts` already has multiple video models with different cost/quality tradeoffs) — a "draft" toggle choosing a cheaper model for the first pass is config, not new infrastructure.

## 3. Showcase area — deepened scoping

Research this pass focused on concrete author/writer portfolio patterns specifically (not general creative portfolios), since GhostWriter's showcase would primarily surface writing-adjacent output.

- **Curation over completeness.** The pattern across writer-portfolio tools (Journo Portfolio, Clippings.me, Authory) is a small curated set — "up to three samples per niche/category" — not a dump of everything a user has built. For GhostWriter this suggests a showcase entry is a deliberately-selected highlight (a project, or even a scene/chapter/production still within a project), not an auto-published feed of every project.
- **Privacy is a first-class, tiered control, not a binary.** Premium portfolio tools gate real privacy features — password-protected pages, unlisted-but-linkable URLs — behind their paid tiers, alongside analytics (view counts). For a paid tool like GhostWriter, this argues against a simple public/private toggle: at minimum, three states are worth considering — private (default), unlisted (shareable link, not indexed/discoverable), and public (indexed, discoverable within the showcase area's own browse/search). This also sidesteps the open question from the first research pass ("visibility model — per-project vs. per-user") somewhat: it can be per-*showcase-entry*, independent of whether the underlying project itself is still private and editable.
- **Format still needs to be mixed-media**, confirming the prior research's ArtStation/Behance-style recommendation over a Wattpad-style linear reader — a showcase entry for a comic-format project needs a page/panel gallery; for a novel, a curated excerpt or blurb; for a production project, stills or the stitched video. This likely means the showcase's data model is a thin "featured content" wrapper around a project's *existing* exports (the CBZ pages, the stitched scene video, the manuscript export) rather than a new content type — reuse of what Export already produces, one more reason this sits naturally on top of already-built architecture rather than requiring new generation pipelines.
- **Still open, genuinely needing the user's decision before any implementation**: whether showcase entries are discoverable by other GhostWriter users at all (a "browse what others built" feed) or purely a personal/shareable-link portfolio with no cross-user discovery — this is a product-positioning choice (community feature vs. personal portfolio tool), not something research can resolve. If a discovery feed is wanted, the Wattpad/Webtoons contest/vote mechanics from the first research pass become relevant; if it's portfolio-only, they don't.

## Synthesis

None of the enhancement or new-feature candidates above require a new architectural pillar — they all extend structures that already exist (capability registry, role split, quality-check stack, World Bible, promise-ledger, Export pipeline). The showcase area remains the one item that's genuinely new and needs the user's product-positioning call (community discovery vs. personal portfolio) before any spec work, let alone code.
