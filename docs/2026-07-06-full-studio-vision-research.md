# Full-studio vision: competitor research + gap analysis against what's already built

Date: 2026-07-06. Companion to the standing product-vision memory (`project-vision-full-studio`) set the same day: *"develop GhostWriter to have a fully production and launch ready pre-production, production and post-production studio for stories, novels, scripts, films, anime, comics, manga and many more entertainment formats — plus an area where people can showcase what they've built, resources, assets, etc."*

This doc researches who else is building pieces of this, then maps every finding against GhostWriter's actual shipped code (not the aspiration) — so the vision becomes a set of scoped, evidenced gaps rather than a mission statement.

## TL;DR

No single competitor covers the full breadth in the vision statement. Every piece has a strong specialized competitor, and GhostWriter's real position is: **best-in-class on the writing/pre-production side, competitive-but-generic on media generation, and starting from zero on the showcase area.** The gaps worth closing, in the order the research supports:

1. **Media-generation quality** (comic/anime consistency, post-production assembly) — GhostWriter's Production/Comic Studios exist and work, but use generic photoreal pipelines where specialized competitors use purpose-built consistency engines. This was already identified in `docs/2026-06-25-ai-director-editor-production-studio-gap-analysis.md`; this research **confirms it's not a stale concern — the entire 2026 anime/comic AI market has converged on "character-lock" as the table-stakes feature**, which raises the cost of continuing to defer it.
2. **Post-production as an actual editing surface**, not just export. GhostWriter's video side stops at ffmpeg-concatenation; competitors treat post as trim/color/sound, a distinct stage with its own UI.
3. **The showcase area** — genuinely unbuilt, no analogous feature exists in the codebase today. This needs its own scope decision (see below), not an incremental panel.
4. **Reverse adaptation** (screenplay→novel, comic→prose) — GhostWriter's Adapt feature is one-directional (novel→screenplay); the closest direct competitor supports both directions plus more formats.

## Where GhostWriter already leads

**Prose generation & world-building** (Sudowrite, NovelAI): Sudowrite is prose-refinement-first (Describe/Rewrite/Expand/Shrink) with a multi-model stack; NovelAI is a sandbox with a Lorebook for world rules. GhostWriter's World Bible (characters/locations/plot-threads/world-entities) + Story Graph + voice fingerprinting + AIisms/humanize detectors together cover more ground than either alone — and unlike both, GhostWriter already has a Director/Writer/Editor role-tagged capability registry (`src/lib/capabilities/registry.ts`) as an architectural spine, which neither competitor's marketing describes having.

**Editorial intelligence** (Ideorix — the closest single competitor by philosophy): Ideorix's Story Bible (Foundation→Plot→Entities, 28 beat structures, 70+ genres), Architect→Writer separation, 13-stage editorial pipeline, 5-agent Beta Reader Panel, and a "Prose Humaniser" blocking 1,212 AI-telltale phrases are all conceptually close to things GhostWriter already ships: `story_plans`/beat sheets (Save-the-Cat/three-act/Hero's-Journey), the Director/Writer/Editor role split (`src/lib/roles/`), `aiisms.ts` (42 fiction tells + structural echo/double-statement checks) and the new `humanize.ts` detector. **GhostWriter's version is smaller in raw count (42 tells vs. 1,212) but architecturally more integrated** — it's wired into generation itself via `MODE_REGISTRY`/`ContextPolicy`, not a separate pass. Ideorix has no comics/anime/film generation at all — confirms this is real whitespace, not a solved problem elsewhere.

**Pre-production stages** (Saga, LTX Studio, StudioBinder/Celtx): these are strong at beat sheets, storyboards, and shot lists for *film specifically*. GhostWriter's beat-sheet/Director tooling already covers the same conceptual ground for prose formats, and — via `generate-package`/Production Studio — already turns a written scene into a shot list + prompts, which is the same "script → storyboard" step these tools sell as a standalone product.

## Where GhostWriter is behind

**Character/style consistency in generated media.** LTX Studio's whole pitch is defining a character once (age/appearance/hairstyle/wardrobe/face) and holding that look across every shot — "directly addresses the character drift problem." Anijam/Toonkit built the same "character-lock" concept for anime specifically. GhostWriter's current mechanism is a single `portraitUrl` reference image passed as `referenceImages` to Segmind/Higgsfield calls (per-shot scoped as of the 2026-06-29 fix logged in CLAUDE.md item 27) — functional, but a single reference image is a weaker consistency mechanism than a structured character-definition object with locked style tokens. **This is the same underlying gap CLAUDE.md item 26 already flagged for Comic Studio** (generic `generateSoulImage()` vs. Segmind's purpose-built `storydiffusion` model) — this research shows it's not comic-specific, it's the standard the whole category has moved to for video and anime too. Recommend treating "structured character-lock" as one shared sub-project across Production Studio, Comic Studio, and any future anime-specific path, rather than three separate comic/video/anime fixes.

**Post-production as a real stage.** Saga's post-production stage explicitly means color grading, transitions, and sound/music integration on a timeline. GhostWriter's equivalent (item 27, CLAUDE.md) is real but narrower: per-shot Seedance generation + `ffmpeg` stream-copy concatenation into one file. There is no re-editable timeline, no trim/reorder-after-generation (Phase C's new filmstrip drag-reorder, shipped today, reorders *before* stitching, not after), no color/sound layer at all. Comic Studio's CBZ export is the equivalent "flatten and ship" step for comics — same shape of gap. **This is a real, evidenced "post-production" hole**, not a vague aspiration — it's the literal difference between "concatenate clips" and "edit a sequence."

**Anime as its own format, not a style preset.** GhostWriter currently treats anime/manga as an `ART_STYLES` preset inside the same photoreal image/video pipeline (`panel-prompt-builder.ts`). Every dedicated anime competitor (Anijam, Toonkit, AnimateAI, DomoAI) runs a materially different model/consistency stack for anime specifically, not a style flag on a generic model. If anime is to be a first-class format per the vision statement, it likely needs its own provider adapter (mirroring how `src/lib/media/adapters/segmind.ts` is already provider-specific), not just another `ART_STYLES` entry.

**Reverse/bidirectional adaptation.** Ideorix converts screenplays (Fountain/FDX/DOCX/PDF) into novels across five length tiers with adjustable POV — GhostWriter's `AdaptPanel` (`docs/superpowers/specs/2026-06-18-adapt-cross-format-design.md`) is explicitly "v1 Novel→Screenplay only," with Web Series/Comic/Higgsfield/Novelization already stubbed as disabled "Coming soon" menu entries. The menu scaffolding already anticipates this; the conversion logic doesn't exist yet.

## The showcase area: no direct precedent, needs its own scope decision

Nothing in GhostWriter's codebase resembles a community/portfolio feature today — confirmed absent, not just unfinished. Researched four analogous platforms for mechanism, not clone-worthy design:

- **Wattpad/WEBTOON**: chapter-reader-first, discovery driven by "audience engagement, community sentiment, data insights," contests where readers vote for favorite creators, personalized feeds + genre category bars.
- **itch.io**: indexed browse/search/recommendation, cover image as the primary discovery hook, tags, explicit creator quality guidelines (a moderation/quality bar, not just an open dump).
- **ArtStation/Behance**: portfolio/case-study format — a "project" is a curated multi-image spread with process notes, not a single linear read. This is the closer analog for GhostWriter, since its output is mixed-media (prose + production stills/video + comic pages), not just text chapters.

**Recommendation, not yet decided by the user:** GhostWriter's showcase needs a project-case-study format (ArtStation/Behance shape) rather than a Wattpad-style linear reader, because a showcased "thing someone built" here is more often a World Bible + a few production stills + a comic page than a full manuscript. Before building anything, this needs an explicit scope decision: what's shown (finished exports only, or in-progress World Bible/assets too?), visibility model (per-project opt-in, presumably — nothing here should be public by default given this is a paid creative tool), and whether it's individual-portfolio-shaped or genre/discovery-feed-shaped. This is a new product surface, not an incremental panel — treat it as its own spec pass when the user is ready to scope it, the same way Adapt and Phase C each got their own design doc before implementation.

## What this changes about backlog priority

The existing backlog (`project-consolidation-roadmap` memory + CLAUDE.md item 26) already had "comic storydiffusion swap," "multi-shot Seedance," and format-specific competitor gaps as separate deferred line items. This research reframes them as **three faces of one underlying gap** (structured character/style consistency across image, video, and eventually anime generation) rather than three unrelated fixes — worth scoping as one shared piece of infra when picked back up, not three sequential patches to three call sites.

## UI/UX research for the vision change (added 2026-07-06, later pass)

The gaps above are almost entirely backend/capability gaps. This pass asks the orthogonal question: as GhostWriter actually grows into "pre-production + production + post-production + showcase," what should the *navigation and screen structure* look like — and does the vision require a new UI paradigm, or does it fit what's already built?

**Headline finding: mostly the latter.** Cross-referencing 2026 UX research (progressive disclosure, command palettes, multi-stage creative-tool workspace design) against GhostWriter's actual shipped UI finds it already follows current best practice in the three places that matter most for this vision, which reframes the open question from "redesign the navigation" to "slot new things into the navigation that already works":

- **Consolidated workspace over scattered screens.** LTX Studio's own UX write-ups highlight its "Projects Workspace" — Gen Space (an experimentation sandbox), Storyboard, and Timeline consolidated into one workspace specifically so "asset management becomes less scattered across different interface sections." GhostWriter already made this exact call: Comic Studio and Production Studio open as overlays inside the single `WritingRoom` shell (the Actions-drawer mechanism, `docs/product-history.md`'s documented fix for the old dual-shell problem) rather than as separate pages you navigate away to. LTX Studio's "sandbox that doesn't affect your main project timeline" concept also already has a direct GhostWriter analog: Phase C's "keep N candidates" (regenerate without overwriting, promote later) is functionally the same idea — worth naming it that way in-product ("try alternates safely") rather than treating it as just an export nicety.
- **Command palette as expected infrastructure, not a nice-to-have.** 2026 SaaS UX research calls Cmd+K "an expected feature" for any single app with an extensive feature set. GhostWriter already has three (`CommandPalette.tsx`, `DashboardCommandPalette.tsx`, `SlashCommandPalette.tsx`) — this is already-correct architecture for a growing tool, not a gap.
- **Progressive disclosure for feature count.** The 2026 pattern is "don't remove features, sequence when users encounter them" (conditional "Advanced" reveals, contextual surfacing). GhostWriter's Density dial (`src/lib/modes/density.ts`'s `DENSITY_META`, Simple/Standard/Full, DB-persisted per-project as of item 38) is exactly this pattern, already built and already the right shape for absorbing new tools (a beta-reader panel or a future anime provider slots into "Standard" or "Full" without needing new chrome).

**Where this DOES have a real implication for the vision:**

- **The showcase area should be a distinct top-level surface, not a project-nested panel** — validated twice now, once by the ArtStation/Behance portfolio-shape research (§3 above) and again by Figma Community's actual structure: a separate `/community` area distinct from the design-file editing surface, with its own gallery/showcase feed and a dedicated "Showcase Your Work" forum category layered on top. GhostWriter's dashboard already has the right home for this shape (`BibleShelf`/`UniverseShelf` shelves, per the 2026-07-03 dashboard redesign) — a `ShowcaseShelf` sitting alongside them is the natural slot, not a new information architecture.
- **The pre/production/post-production framing should map onto, not replace, the existing stage ladder.** GhostWriter already has two overlapping stage taxonomies doing real work: the 5-stage "One Path" ladder (Idea→Structure→Draft→Polish→Export) and the capability registry's 4-stage funnel (discover/shape/write/produce, `src/lib/capabilities/registry.ts`'s `CapabilityStage`). Introducing a third, vision-specific "pre-production/production/post-production" navigation layer would directly contradict the progressive-disclosure principle this research just validated as already correctly applied. The honest mapping: pre-production ≈ discover+shape (World Bible, Story Graph, beat sheets), production ≈ write+produce's generation half (Draft stage, Production/Comic Studio generation), post-production ≈ produce's assembly half (Polish stage, Phase C review, export/stitching). The vision statement is a marketing/positioning frame for work already organized this way — not a request for new chrome.
